## 🔹 01. Docker 이미지(Image)와 컨테이너(Container)의 차이를 설명해보세요.

Docker는 애플리케이션과 실행 환경을 함께 패키징하여 **어디서나 동일하게 실행**할 수 있도록 하는 컨테이너 기반 기술입니다.
이때 핵심 개념이 바로 **Image**와 **Container**입니다.

---

### 📌 Docker Image란?

> **애플리케이션 실행에 필요한 모든 것을 포함한 ‘불변(Immutable) 템플릿’**

Docker 이미지는 다음 내용을 포함합니다.

* 애플리케이션 코드
* 런타임 환경 (Python, Node, Java 등)
* 라이브러리 및 의존성
* 실행 명령 (CMD, ENTRYPOINT)

이미지는 **실행되지 않으며**, 어디까지나 “설계도” 또는 “스냅샷”의 개념입니다.
Dockerfile을 기반으로 한 번 빌드되면 내용이 변경되지 않습니다.

---

### 📌 Docker Container란?

> **이미지를 실행한 ‘살아있는 인스턴스’**

컨테이너는 Docker 이미지를 실행(run)한 결과로 생성되며,

* 실제로 프로세스가 동작하고
* 메모리, CPU, 네트워크를 사용하며
* 생성·중지·삭제가 가능

즉, 컨테이너는 **이미지의 실행 상태 객체**입니다.

---

### 📌 Image vs Container 비교

| 구분 | Docker Image  | Docker Container |
| -- | ------------- | ---------------- |
| 성격 | 정적인 템플릿       | 동적인 실행 인스턴스      |
| 상태 | 불변(Immutable) | 가변(Stateful)     |
| 실행 | 실행되지 않음       | 실제로 실행됨          |
| 역할 | “무엇을 실행할지” 정의 | “실제로 실행 중인 것”    |
| 비유 | 클래스(Class)    | 객체(Instance)     |

---

### ✅ 핵심 정리

> **Image는 ‘어떻게 실행할지’이고,
> Container는 ‘실제로 실행된 결과’입니다.**

## 🔹 02. Docker Compose는 어떤 상황에서 유용하며, Dockerfile과 어떤 차이가 있나요?

현실적인 서비스는 하나의 컨테이너로만 구성되지 않습니다.
예를 들어 다음과 같은 조합이 흔합니다.

* Backend (API 서버)
* Database (MySQL, Redis)
* Frontend (React, Nginx)

이런 **다중 컨테이너 환경**을 효율적으로 관리하기 위해 등장한 도구가 **Docker Compose**입니다.

---

### 📌 Dockerfile의 역할

> **“컨테이너 하나를 어떻게 만들 것인가”를 정의**

* 단일 이미지 빌드에 집중
* 애플리케이션 실행 환경을 코드로 정의
* `docker build` → `docker run` 흐름

```dockerfile
FROM python:3.10
COPY . /app
WORKDIR /app
RUN pip install -r requirements.txt
CMD ["python", "main.py"]
```

---

### 📌 Docker Compose의 역할

> **“여러 컨테이너를 어떻게 함께 실행할 것인가”를 정의**

* 여러 서비스(API, DB, Cache 등)를 **한 번에 관리**
* 네트워크, 볼륨, 실행 순서까지 통합 정의
* `docker-compose up` 한 줄로 전체 실행

```yaml
services:
  api:
    build: .
    ports:
      - "8000:8000"
  redis:
    image: redis:latest
```

---

### 📌 Dockerfile vs Docker Compose 비교

| 구분    | Dockerfile         | Docker Compose      |
| ----- | ------------------ | ------------------- |
| 목적    | 이미지 생성             | 다중 컨테이너 실행 관리       |
| 단위    | 하나의 컨테이너           | 여러 컨테이너             |
| 주요 역할 | 환경/의존성 정의          | 서비스 연결 및 오케스트레이션    |
| 실행    | `docker build/run` | `docker-compose up` |
| 사용 시점 | 빌드 단계              | 실행/배포 단계            |

---

### ✅ 언제 Compose를 쓰는가?

* 로컬 개발 환경에서 **백엔드 + DB**를 함께 띄울 때
* 팀원 간 실행 환경을 동일하게 맞추고 싶을 때
* CI/CD 이전 단계에서 간단한 서비스 묶음을 관리할 때

## 🔹 03. Kubernetes의 기본 개념은 무엇이며, Docker와는 어떤 관계가 있나요?

Docker가 **“컨테이너를 만드는 기술”** 이라면,
Kubernetes는 **“컨테이너를 운영하는 기술”** 입니다.

---

### 📌 Kubernetes란?

> **컨테이너 오케스트레이션(Orchestration) 시스템**

Kubernetes는 다음과 같은 문제를 해결합니다.

* 컨테이너가 수십~수백 개로 늘어났을 때
* 서버가 죽으면 컨테이너를 자동으로 복구하고 싶을 때
* 트래픽 증가에 따라 자동 확장을 하고 싶을 때

---

### 📌 Kubernetes의 핵심 개념

| 개념             | 역할                             |
| -------------- | ------------------------------ |
| **Pod**        | 컨테이너의 최소 실행 단위 (1개 이상 컨테이너 포함) |
| **Node**       | Pod가 실행되는 실제 서버                |
| **Cluster**    | 여러 Node의 집합                    |
| **Deployment** | Pod 배포 및 복제(Scaling) 관리        |
| **Service**    | Pod에 대한 안정적인 네트워크 접근 제공        |

---

### 📌 Docker와 Kubernetes의 관계

> **Docker는 ‘빌드 & 실행’,
> Kubernetes는 ‘관리 & 자동화’**

| 구분    | Docker            | Kubernetes    |
| ----- | ----------------- | ------------- |
| 역할    | 컨테이너 생성 및 실행      | 컨테이너 운영·관리    |
| 책임 범위 | 로컬/단일 환경 중심       | 대규모·분산 시스템    |
| 관계    | Kubernetes의 기반 기술 | Docker 위에서 동작 |

※ 현재 Kubernetes는 Docker뿐 아니라 **containerd, CRI-O** 같은 런타임도 사용하지만,
**컨테이너 개념 자체는 Docker에서 시작**되었다는 점이 중요합니다.

---

### ✅ 전체 흐름 정리

```text
Dockerfile → Docker Image → Docker Container
           ↓
     Docker Compose (로컬 다중 컨테이너)
           ↓
     Kubernetes (대규모 운영 자동화)
```

---

## ✅ 최종 요약

* **Docker Image**는 실행 설계도, **Container**는 실제 실행 객체입니다.
* **Dockerfile**은 단일 컨테이너 정의, **Docker Compose**는 다중 컨테이너 실행 관리 도구입니다.
* **Kubernetes**는 Docker 컨테이너를 대규모로 안정적으로 운영하기 위한 오케스트레이션 시스템입니다.
